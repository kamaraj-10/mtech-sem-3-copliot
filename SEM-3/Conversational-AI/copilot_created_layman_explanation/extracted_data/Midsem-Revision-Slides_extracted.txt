BITS Pilani, Deemed to be University under Section 3 of UGC Act, 1956Hands-On: Byte-Pair Encoding•BPEis a "subword" tokenization algorithm. It sits in the "Goldilocks" zone between character-based tokenization (too long, little meaning) and word-based tokenization (huge vocabulary, fails on unknown words)How it works (The Concept):1.Training:It reads a massive corpus of text and counts adjacent pairs of characters.2.Merging:It takes the most frequent pair (e.g., 'e' and 'r' -> 'er') and adds it to the vocabulary as a new unit.3.Iterating:It repeats this thousands of times. Eventually, it learns common prefixes (un-,re-), suffixes (-ing,-ed), and whole common words (apple,table).BPE training starts by computing the unique set of words used in the corpus (after the normalization and pre-tokenization steps are completed), then building the vocabulary by taking all the symbols used to write those words. As a very simple example, let’s say our corpus uses these five words:"hug", "pug", "pun", "bun", "hugs”The base vocabulary will then be["b", "g", "h", "n", "p", "s", "u"].

BITS Pilani, Deemed to be University under Section 3 of UGC Act, 1956Byte-Pair Encoding-first merginglet’s assume the words had the following frequencies:("hug", 10), ("pug", 5), ("pun", 12), ("bun", 4), ("hugs", 5)meaning"hug"was present 10 times in the corpus,"pug"5 times,"pun"12 times,"bun"4 times, and"hugs"5 times.•We start the training by splitting each word into characters (the ones that form our initial vocabulary) so we can see each word as a list of tokens:•("h" "u" "g", 10), ("p" "u" "g", 5), ("p" "u" "n", 12), ("b" "u" "n", 4), ("h" "u" "g" "s", 5)•Then we look at pairs. The pair(”u", ”g")is present in the words"hug","pug", and"hugs", for a grand total of 20 times in the vocabulary.

BITS Pilani, Deemed to be University under Section 3 of UGC Act, 1956Byte-Pair Encoding-Second MergingThus, the first merge rule learned by the tokenizer is("u", "g") -> "ug", which means that"ug"will be added to the vocabulary, and the pair should be merged in all the words of the corpus. At the end of this stage, the vocabulary and corpus look like this:Vocabulary: ["b", "g", "h", "n", "p", "s", "u", "ug"]Corpus: ("h" "ug", 10), ("p" "ug", 5), ("p" "u" "n", 12), ("b" "u" "n", 4), ("h" "ug" "s", 5)Now we have some pairs that result in a token longer than two characters: the pair("h", "ug"), for instance (present 15 times in the corpus). The most frequent pair at this stage is("u", "n"), however, present 16 times in the corpus, so the second merge rule learned is("u", "n") -> "un".Adding that to the vocabulary and merging all existing occurrences leads us to:Vocabulary: ["b", "g", "h", "n", "p", "s", "u", "ug", "un"]Corpus: ("h" "ug", 10), ("p" "ug", 5), ("p" "un", 12), ("b" "un", 4), ("h" "ug" "s", 5)

BITS Pilani, Deemed to be University under Section 3 of UGC Act, 1956Byte-Pair Encoding-Third MergingNow the most frequent pair is("h", "ug"), so we learn the merge rule("h", "ug") -> "hug", which gives us our first three-letter token. After the merge, the corpus looks like this:Vocabulary: ["b", "g", "h", "n", "p", "s", "u", "ug", "un", "hug"]Corpus: ("hug", 10), ("p" "ug", 5), ("p" "un", 12), ("b" "un", 4), ("hug" "s", 5)And we continue like this until we reach the desired vocabulary size.Let’s take the example we used during training, with the three merge rules learned:("u", "g") -> "ug"("u", "n") -> "un"("h", "ug") -> "hug”The word"bug"will be tokenized as["b", "ug"]."mug", however, will be tokenized as["[UNK]", "ug"]since the letter"m"was not in the base vocabulary. Likewise, the word"thug"will be tokenized as["[UNK]", "hug"]: the letter"t"is not in the base vocabulary, and applying the merge rules results first in"u"and"g"being merged and then"h"and"ug"being merged.How do you think the word"unhug"will be tokenized?

BITS Pilani, Deemed to be University under Section 3 of UGC Act, 1956Why Do We Need KV-Cache?Understanding Attention Computation in Autoregressive GenerationThe Inefficiency Problem: During text generation, transformers generate one token at a time. Each new token must attend to ALL previous tokens —recomputing K, V from scratch is O(n²) total work.Without KV-Cache (Naïve)Token 1: "The" → Compute Q₁, K₁, V₁Token 2: "cat" → Compute Q₁,K₁,V₁, Q₂,K₂,V₂  (recompute!)Token 3: "is" → Compute Q₁,K₁,V₁, Q₂,K₂,V₂ , Q3,K3,V3 Recompute all Q,K,V againToken n: Recompute everything for tokens 1 to n-1✗Cost grows as O(n²) for n tokens!With KV-Cache (Optimised)Token 1: "The" → Compute K₁, V₁ → Cache themToken 2: "cat" → Reuse K₁,V₁; compute K₂,V₂ → CacheToken 3: "is" → Reuse K₁,K₂,V₁,V₂; compute K₃,V₃Token n: Reuse all cached K,V —compute only Kₙ,Vₙ✓Only compute new K,V → O(n) total!Why Only Cache K and V?Query (Q): Changes with each new token position —must recompute. Keys (K) & Values (V): Represent previous context —constant once computed. Cache them once, reuse forever.1 op per token, "op" looselyto mean"process one token's K and V together"—it's a counting unit for comparison, not a literal single matrix multiply.

1. Iteration LimitsProblem: Agent gets stuck in reasoning loopSolution: Set max iterations (typically 10). If limit reached, log history, return error to user, alert monitoring system.
2. Repetition DetectionProblem: Agent calls the same failed tool repeatedlySolution: Track action signatures (tool name + args hash). If same action appears 3×in history → force completion or try alternative.
3. Tool Failure TrackingProblem: Problematic tool causes repeated failuresSolution: Track failures per tool using defaultdict(int). After 3 consecutive failures, remove from available tools for this session.
4. Timeout ProtecTonProblem: Agent takes too long, tying up resourcesSolution: Implement 60-second alarm using signal module (Unix) or threading.Timer (cross-platform). Raise TimeoutError when exceeded.
5. Input ValidationProblem: Invalid function arguments cause crashesSolution: Use Pydantic models for all function parameters. Validate before execution. If validation fails, return structured error to model for retry.
6. Graceful DegradationProblem: Tool unavailable but task still needs completionSolution: When tool fails, provide alternative suggestions. Maintain fallback tools or guide model to alternative approaches.

BITS Pilani, Deemed to be University under Section 3 of UGC Act, 1956LoRA: Low-Rank Adaptation  (Hu et al., 2021)
Key InsightNeural networks contain many dense layers performing matrix multiplication. Inspired by the intrinsic dimension hypothesis, the weight updates ΔW during fine-tuning have low intrinsic rank.
Pre-trained matrix:W₀ ∈ℝ^(d×k) —frozen during training
Updated matrix:W₀ + ΔW = W₀ + BA
Low-rank matrices:A ∈ℝ^(r×k),  B ∈ℝ^(d×r),  where r ≪min(d,k)
Hidden state:h = W₀x  →  h = W₀x + ΔWx = W₀x + BAx
Rank r:Typically 2–64; r=4 or r=8 already gives strong resultsHu, Edward J., et al. "LoRA: Low-Rank Adaptation of Large Language Models." ICLR 2022.

Motivation: 32-bit floating-point computations are memory-intensive. Quantization reduces the precision of numbers used in the model, dramatically cutting memory.
int8 tensorRange: [−127, 127]  (16 levels)
int4 tensorRange: [−8, 7]  (16 levels, signed)
float32Range: ~1.18×10⁻³⁸ to 3.4×10³⁸
Quantization / Dequantization FormulasQuantize:  X[int8] = round( (127 / absmax(X[FP32])) ×X[FP32] )Dequantize:  dequant(X[FP32], X[int8]) = X[int8] / (127 / absmax(X[FP32]))
⚠Quantization with outliers leads to larger quantization errors.

BITS Pilani, Deemed to be University under Section 3 of UGC Act, 19564-bit NormalFloat(NF4)  (Dettmers et al., 2023)•Neural network weights typically follow a zero-centered normal distribution N(0, σ²)•NF4 is an information-theoretically optimal quantization type for normally distributed data•Normalization: transform all weights to a single fixed distribution by scaling with σ and setting range to [−1, 1] with mean 0 and standard deviation 1•Quantize the normalized weights to 4 bits —16 levels evenly spaced within the normalized values•NF4 yields better empirical results than both 4-bit integers and 4-bit floats•Small dequantization error: for example, 0.1832 rounds to 0.2 → error ≈ 0.0168
NF4 (NormalFloat)Information-theoretically optimal for normal distributions. Used in QLoRA.
int4 (Integer)Uniformly spaced integers. Not optimal when weight distribution has thin tails.
fp4 (Float)4-bit floats —fewer levels near zero where weights cluster.Dettmers, Tim, et al. "QLoRA: Efficient Finetuning of Quantized LLMs." NeurIPS 2023. https://doi.org/10.48550/arXiv.2305.14314

BITS Pilani, Deemed to be University under Section 3 of UGC Act, 1956QLoRA: Quantization + Low-Rank Adapters  (Dettmers et al., 2023)
QLoRA combines 4-bit NF4 quantization with Low-Rank Adapters to enable efficient fine-tuning of large language models on a single consumer GPU.
1. 4-bit NormalFloatPre-trained model weights quantized to NF4 and frozen. Significantly reduces memory usage at load time.
2. Double QuantizationQuantize the quantization constants themselves, saving ~0.37 bits/parameter (≈3 GB for a 65B model).
3. Paged OptimizersNVIDIA unified memory to avoid gradient checkpointing spikes when processing long-sequence mini-batches.
Memory reduction: Full FT of 65B model requires >780 GB GPU memory.  QLoRA trains same model in <48 GBwithout degrading performance vs. 16-bit fully fine-tuned baseline.

BITS Pilani, Deemed to be University under Section 3 of UGC Act, 1956QLoRAArchitecture & Training Flow
1Load Base ModelLoad pre-trained LLM (e.g. LLaMA 70B) in 4-bit NF4 format. Weights are frozen —no gradient flows through them.
2Add LoRA AdaptersAttach small trainable rank-r matrices A and B to each transformer layer (attention Wq, Wv). Adapters are in full 32-bit precision.
3Forward PassInput passes through quantized (frozen) base weights + full-precision LoRA adapters. Output = W₀x + BAx.
4Backward PassGradients flow only through LoRA matrices A and B. Quantized weights dequantized temporarily for computation; requantized after.
5Final ModelConsists of original 4-bit weights + trained adapter matrices in bfloat16/fp32. Can merge adapters into base model for zero-latency inference.Dettmers, Tim, et al. "QLoRA: Efficient Finetuning of Quantized LLMs." NeurIPS 2023.
